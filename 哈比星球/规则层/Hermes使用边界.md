# Hermes 使用边界

## 一、Hermes 在本项目中的正确位置

Hermes 是：

- 运行时
- 执行承载层
- 工具调用层
- 留痕层

Hermes 不是：

- 项目事实定义者
- 项目制度定义者
- 最终发布决策者

## 二、Hermes 底层能力的正确用法

### 1. Profile

用途：

- 隔离岗位状态
- 隔离会话
- 隔离技能加载

边界：

- 不是权限沙箱
- 不是组织治理系统

### 2. Sessions / state.db / logs

用途：

- 追踪任务执行痕迹
- 追踪问题来源
- 追踪岗位动作

边界：

- 留痕不等于合规
- 有记录不等于结论正确

### 3. Skills

用途：

- 复用稳定工作方法
- 把重复动作模板化

边界：

- 技能必须岗位化
- 不允许演化成无限人格技能树

### 4. delegate_task

用途：

- 短链拆分
- 并行查证
- 临时草拟

边界：

- 不作为长链主流程
- 不作为治理主流程
- 不作为审核闭环主流程

### 5. Tooling

用途：

- 文件处理
- 搜索
- 终端操作
- MCP 能力扩展
- 自动化

边界：

- 工具能力不能替代岗位边界
- 会做不等于有权做

## 三、最小稳定原则

本项目固定遵守：

1. 少岗位
2. 窄职责
3. 强模板
4. 强回查
5. 强审核

## 四、判定标准

如果某种做法让 Hermes 开始：

- 自行扩张职责
- 自行定义规则
- 自行定义事实
- 混淆调度 / 执行 / 审核

则该做法一律视为越界。
