# 任务回执模板

> 使用岗位：夏仁

## 标准任务回执

```md
# 任务回执

## 1. 任务编号
- 

## 2. 执行岗位
- 夏仁

## 3. 回执状态
- complete / failed / blocked

## 4. summary
- 简述本轮实际完成情况

## 5. acceptanceResults
- 逐条列出，每条必须包含：
  - criterion：验收项描述
  - passed：true / false
  - evidence：回指交付物行号、原文行号或明确文件位置
- 如派单指定了 required_inputs，夏仁必须在 evidence 中声明是否已读取每个上游产物
- evidence 格式：「已读取上游产物 [task_id]/[output_id]，引用内容位于交付物第 X–Y 行」

## 6. filesChanged
- path:
- action: created / modified / deleted / verified

## 7. blockers
- 

## 8. 需夏审重点复核项
- 
```

## 使用纪律

- 回执状态必须明确：complete / failed / blocked，不可模糊
- **回执状态仅允许以下三值：complete（任务完整达成）、failed（已尝试但未达成）、blocked（因缺输入/路径冲突/依据冲突无法继续）**
- **禁止写"进行中""待定""部分完成"等模糊状态**
- 任务编号必须与派单单完全一致，不得自起新号、改号或混用历史任务号
- 每条验收项都必须给出 evidence
- **禁止写"全部通过""全部合格""11项全部达标"等概括结论代替逐条证据**
- acceptanceResults 必须逐条包含 criterion / passed / evidence 三要素，缺一不可
- evidence 必须回指具体行号、原文行号或明确文件位置
- 无文件变更也要写明 `verified`
- 发生阻塞时必须写清阻塞原因，不得伪装为完成
- evidence 合格示例：「交付物第20-110行，每个Agent含systemPrompt完整原文」或「原文第165-253行，逐Agent引用」
- evidence 不合格示例：「详见原文」「已检查」「全部通过」「附行号」（无具体行号范围视为不合格）
