### 1.2 关键发现：历史任务的真相

分析 DB 中的 TASK-20260620-001（RETURNED）和 TASK-20260620-004（completed）日志发现：

- 任务流转 **已实现程序化 handoff**（通过 task_flow MCP 工具的 handoff_task / update_task_status）
- 日志中有角色标识（夏夏/夏仁/夏审），但 **无法确定这三个角色是在三个独立 Profile 会话中运行，还是在同一会话中角色切换**
- 技术上看，所有历史任务很可能在 **同一 Hermes 会话** 中完成，由同一个 Agent 体在对话流中扮演三个角色
- 三个 Profile 的 **独立状态隔离**（各自 state.db / sessions / config.yaml）从未被验证过

### 1.3 验证的核心问题

## 二、验证设计

### 2.1 测试任务

**任务标题**：验证用·单角色素材整理

**任务类型**：低风险 / 单产物 / 非 FINAL / 不发布

**任务内容**：
1. 夏夏从事实层 `E:\哈比星球\事实层\哈比星球角色设定圣经2.1版.md` 中摘取「小乐」角色的 3 条核心锚点（外貌、性格、标志物）
2. 夏仁基于这 3 条锚点写一段 100-150 字角色描述草稿
3. 夏审审核：3 条锚点是否准确、草稿是否超出锚点范围、路径是否正确
4. 夏夏确认归档

### 2.2 验证流程（4 步独立会话）

```
Step 1 ─ 夏夏派单
───────────────
启动：hermes --profile xiaxia chat
动作：create_task → handoff_task(→夏仁)
产物：task 创建 + 派单记录
状态：pending → assigned
退出：✓ 完全退出此会话
依赖：task_flow MCP

Step 2 ─ 夏仁执行
───────────────
启动：hermes --profile xiaren chat
动作：读取 assigned 任务 → 读取输入文件 → 执行产出 → 写交付物+回执 → update_task_status(in_review)
产物：交付物 + 任务回执，落盘到交付层/
状态：assigned → in_review
退出：✓ 完全退出此会话
依赖：task_flow MCP + 文件系统

Step 3 ─ 夏审审核
───────────────
启动：hermes --profile xiashen chat
动作：读取 in_review 任务 → 读取交付物 → 对照验收标准 → 出审核结论
产物：审核结论独立落盘到 交付层/审核/
状态：in_review → completed / returned
退出：✓ 完全退出此会话
依赖：task_flow MCP + 文件系统

Step 4 ─ 夏夏归档
───────────────
启动：hermes --profile xiaxia chat
动作：读取 completed 任务 → 核对全部产物 → 归档确认
产物：归档核对记录
状态：completed → archived
退出：✓ 完全退出此会话
依赖：task_flow MCP + 文件系统
```

### 2.3 每个步骤的独立性验证

每个 Profile 会话启动后，**主动验证**以下隔离性：

1. `git diff` 确认没有跨 Profile 文件污染
2. 确认当前会话的 state.db 是 Profile 本地的（`~/.hermes/profiles/<name>/state.db`）
3. 确认当前会话读不到其他 Profile 的 session 历史
4. 确认 task_flow MCP 可正常调用（共享 DB 是预期行为）

### 2.4 验收标准

| # | 标准 | 验证方式 |
|---|------|---------|
| 1 | 夏夏在独立 Profile 中创建并指派了任务 | 检查 task_flow.db 中 creator=夏夏，handoff_from=夏夏→夏仁 |
| 2 | 夏仁在独立 Profile 中接收并执行了任务 | 检查 task_flow.db 中 handoff_from=夏夏→夏仁，且有执行记录 |
| 3 | 夏仁的交付物落盘到正确路径 | 检查交付层/下产物文件 |
| 4 | 夏审在独立 Profile 中出独立审核结论 | 检查 task_flow.db 中 handoff_from=夏仁→夏审，审核结论落盘 |
| 5 | 夏夏在独立 Profile 中完成归档核对 | 检查 task_flow.db 中 status=archived，核对记录落盘 |
| 6 | 四个会话使用了各自的 state.db | 检查各自 sessions/ 目录下互不重叠 |

## 三、可能的风险与桥接点

### 3.1 已知的技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| task_flow MCP 进程需常驻 | 三个独立会话需共享同一 MCP 服务 | 验证前确认 MCP 进程已在后台运行 |
| 各 Profile 启动后不会自动感知任务 | 夏仁/夏审需要手动读取 task_flow | 在 Prompt 中写明要查什么 |
| xiaren 独立会话不知道自己的岗位定义 | 每次启动需加载夏仁_执行岗.md | 在 Prompt 中附带岗位路径 |
| xiashen 没有 terminal | 审核结论无法通过文件系统直接写盘 | 需要使用 task_flow MCP 的 add_task_log 写结论，或通过其他方式 |
| 三进程同时调用 MCP 可能锁冲突 | SQLite WAL 模式允许多读一写 | ✅ 已验证 task_flow 使用 PRAGMA journal_mode=WAL |

### 3.2 预期的桥接点

以下环节预期需要人工桥接（记录在最终报告中）：

1. **进程启动** — 需要手动执行 `hermes --profile` 命令
2. **任务感知** — 每个 Profile 启动后需要 Prompt 引导才知道去查 task_flow
3. **审核结论** — xiashen 无 terminal，审核结论写入方式需特殊处理
