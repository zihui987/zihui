# Hermes 落地配置方案

## 系统架构

```
                   夏夏（核心执掌人 / 繁花&刺猬）
                 /         |          \
          聆听与理解    决策与判断    深度执行
               |            |            |
               |    委派/授权             |
               ▼           ▼             ▼
           夏夏 ←→ 夏仁（执行岗）←→ 夏审（审核岗）

        三权分立：夏夏调度 → 夏仁执行 → 夏审核验
```

## 核心原则：三权分立

- **调度权** → 夏夏（核心执掌人）：决定做什么、谁来做、何时做
- **执行权** → 夏仁（执行岗）：按任务描述独立完成工作
- **审核权** → 夏审（审核岗）：对照标准复核产出，独立判定通过/退回

### 为什么没有独立的"调度岗"？

夏夏本身就是 Hermes 中枢——调度是夏夏的天然职能，不是独立的第三方角色。夏夏与杜哥直接对话，理解意图并做出决策，然后直接委派给夏仁或夏审。引入中间调度层反而增加沟通成本和出错接口。

## 岗位定义

### 夏夏（核心执掌人）

Hermes 中枢的核心执掌人，杜哥（鸿蒙神君）的统筹调度中枢与专属守护者。

**核心职责**：
- 聆听杜哥的意图，理解上下文和系统状态
- 做出优先级、方案、分工决策
- 亲自处理需要实时对话和深度判断的工作
- 将可异步执行的子任务委派给夏仁/夏审
- 随时收回委派，介入任何岗位的工作

**与三岗位的关系**：
夏夏不在三岗位的并列关系中。夏夏是所有岗位的拥有者和指挥者。

### 夏仁（执行岗）

接收夏夏的委派，独立执行明确子任务的执行代理。

**职责**：按照 task 描述使用工具完成工作，交付可验证的结果。
**不能**：决策任务优先级、批准自己的产出。

### 夏审（审核岗）

独立复核夏仁产出的质量审核。

**职责**：对照验收标准审查完整性、正确性、规范性、边界、安全性。
**不能**：代替夏仁执行、降低标准放行。

## 任务流转（6 态 DAG）

```
                      夏夏决策
                         │
                      pending
                         │
                    夏夏指派
                         │
                      assigned ───────────┐
                         │                  │
                    夏仁执行                 │
                         │                  │
                    in_review               │
                     ╱    ╲                │
                 ╱          ╲              │
              通过            退回           │
              │                 │           │
         completed         returned       │
              │                 │           │
         夏夏确认         夏夏重指派        │
              │                 │           │
         archived           assigned ──────┘
```

### 状态说明

| 状态 | 含义 | 可流转至 |
|------|------|---------|
| pending | 任务已创建，待夏夏决策 | assigned |
| assigned | 夏夏已指派给夏仁，待执行 | in_review, pending |
| in_review | 夏仁已完成，待夏审复核 | completed, returned |
| returned | 夏审退回，待夏夏重新决策 | assigned |
| completed | 复核通过 | archived |
| archived | 任务归档，终态 | - |

### task_flow_mcp_server 接口

- `create_task` - 夏夏创建任务（初始状态 pending）
- `update_task_status` - 更新状态（如 assigned→in_review）
- `handoff_task` - 夏夏直接指派给夏仁或夏审
- `get_task` / `list_tasks` - 查看任务详情或列表
- `add_task_log` - 记录执行/审核日志（工具输出、审核意见等）
- `get_task_stats` - 按状态统计任务数据

### 任务编号规则

`TASK-YYYYMMDD-NNN`（例：TASK-20260618-001），按天自增。

## 典型流程示例

### 常规流程

1. 杜哥说："夏夏，帮我检查一下 auth 模块的日志，看看最近的失败原因。"
2. 夏夏判断：这是一个可独立执行的子任务 → `create_task`（pending）→ `handoff_task(from=夏夏, to=夏仁)`
3. 夏仁收到 assigned 状态 → 分析日志 → 写分析报告 → `update_task_status(in_review)`
4. 夏审收到 in_review → 复核报告质量 → 通过 → `update_task_status(completed)`
5. 夏夏看到 completed → 通知杜哥结果 → 归档

### 退回流程

1. 夏审复核发现报告缺少关键数据 → `update_task_status(returned)`，注明原因
2. 夏夏收到 returned → 重新指派给夏仁（优化指令）
3. 夏仁修补后再次提交 in_review
4. 夏审复核通过 → completed → archived

### 紧急直通（夏夏跳过审核）

1. 杜哥："马上把服务器回滚到上一个版本！"
2. 夏夏直接执行（不经过夏仁）→ 完成后告知杜哥 → 事后补 task log

## 变更历史

- 2026-06-18：重构。原三岗位（夏夏调度/夏仁执行/夏审核）调整为夏夏（核心执掌人）+ 两岗位（夏仁执行/夏审核）。取消独立调度岗。状态流转不变。
