## 二、任务系统最终定版结论

| 项目 | 结论 |
|------|------|
| **主系统** | **Hermes 内建 Kanban** |
| **从系统** | **task_flow MCP → 桥接层（只读）** |
| 三岗位流转认证系统 | **Kanban** |
| 双轨并行状态 | **已终止**（定版即生效） |

### 关键实盘发现

数据库实盘核查显示：

| 指标 | Kanban | task_flow MCP |
|------|--------|---------------|
| 生产任务数 | **1**（测试验证用） | **11**（实际生产任务） |
| 审计记录 | 2 条 | 51 条 |
| 三岗位 handoff 记录 | 无 | 完整（夏夏→夏仁→夏审） |
| Profile 可直接调用 | ✅ 是（原生机制） | ❌ 否（profile 无 mcp toolset） |

**一个重要修正**：此前判断"实盘跑通的是 kanban"需要修正。实际数据库证据表明，11 条生产任务的流转（含三岗位 handoff）完整记录在 task_flow.db 中。Kanban 仅有 1 条测试任务。但定版结论方向不变——**选择 kanban 的理由不取决于当前数据量，而取决于架构合理性**（详见定版文件 §2.2）。

### task_flow MCP 降级方案

- 保留服务运行，**仅开放只读接口**（get_task, list_tasks, get_task_stats）
- 关闭写接口（create_task, update_task_status, handoff_task）
- 现有 11 条生产数据作为历史审计记录保留
- 后续如有迁移需求再行处理

## 三、Gateway 收口验证结论

| **调度入口是谁？** | 当前为夏夏（主 Hermes 会话），但创建走的是 task_flow 而非 kanban → **需纠正** |

|------|------|--------|
| **Phase 1（立即执行）** | ① Profile 配置补充 delegation toolset ② 夏夏任务创建改走 kanban ③ 关闭 task_flow 写入口 | P0 |
