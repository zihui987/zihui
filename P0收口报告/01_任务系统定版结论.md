## 2. 定版依据

### 2.1 数据库实盘验证结论

| 维度 | Kanban | task_flow MCP |
|------|--------|---------------|
| 任务数 | 1 条（测试用） | 11 条（生产任务） |
| 任务ID格式 | `t_822e819a`（随机hash） | `TASK-YYYYMMDD-NNN`（可读编号） |
| 状态流转 | ready → completed | pending → assigned → in_review → completed → archived |
| 审计日志 | task_events 表（2条记录） | task_logs 表（51条记录） |
| 角色支持 | assignee + worker_pid | assignee + handoff_from/handoff_to |
| 实际使用 | 1次验证测试 | 完整三岗位生产流转 |
| 数据时间范围 | 仅 2026-06-20 | 2026-06-18 ~ 2026-06-20 |

### 2.2 为什么主系统定为 Kanban

1. **Hermes 原生机制**：Kanban 是 Hermes 内建的任务调度系统，与 gateway、profile、worker 体系深度集成
2. **Profile 即 worker**：三个岗位（xiaxia/xiaren/xiashen）已注册为 Hermes profile，天然适配 kanban dispatch
3. **自动调度基础设施**：`dispatch_in_gateway: true` + `dispatch_interval_seconds: 60` + `auto_decompose: true` 已配置就绪
4. **Worker 生命周期管理**：Kanban 原生支持 worker_pid、consecutive_failures、claim_lock、max_runtime_seconds 等生产级能力
5. **配置一致性**：主 config.yaml 的 kanban 段已经定义了 dispatch 策略

### 2.3 为什么 task_flow MCP 不再承担主流转

1. **Profile 无权调用**：三个 profile 的 config.yaml 均未包含 `mcp` toolset，无法直接调用 task_flow MCP 工具
2. **手动串联现状**：task_flow 在实际运行中作为数据存储层（审计日志+状态记录），所有状态更新由 夏夏 手动写入，并非自动流转
3. **双轨风险**：继续双轨并行会导致任务状态分裂——部分在 kanban.db，部分在 task_flow.db，审计链无法统一
4. **非原生能力**：task_flow MCP 是自定义 Python 服务，不具备 Hermes 原生 gateway dispatch 的 worker 管理、重试、failure circuit breaker 等能力

---

## 3. task_flow MCP 后续定位：三选一

### 选定：桥接层 — 作为 Kanban 的观察/桥接通道

**理由**：
- task_flow 已有 11 条生产任务数据和 51 条审计记录，不宜直接退役
- task_flow 的 `TASK-YYYYMMDD-NNN` 编号体系适合外部引用
- task_flow 的详细 handoff 日志可作为 kanban 事件的补充视图

**具体做法**：
- 保留 task_flow MCP 服务运行，但**停止用于任务状态变更**
- task_flow 只作为只读查询层：`get_task`、`list_tasks`、`get_task_stats`
- 所有新任务创建、状态流转统一走 kanban
- 未来视情况考虑将 task_flow 数据迁移至 kanban 后退役

**不选择的方案**：
- ~~观察层~~：只读不写，与现状差异不大，无法解决双轨问题
- ~~退役候选~~：当前 task_flow 中有未迁移的生产数据，硬退役会导致丢失

## 4. 定版后规则

| 操作 | 归属系统 | 说明 |
|------|----------|------|
| 任务创建 | **Kanban** | 夏夏通过 kanban 创建任务 |
| 任务派发 | **Kanban dispatch** | gateway 按 dispatch_interval 派发至 profile worker |
| 任务执行 | **Profile worker** | 夏仁/夏审 在各自的 profile session 中执行 |
| 状态更新 | **Kanban** | 状态由 worker 或 夏夏 更新至 kanban |
| 审计记录 | **Kanban event** | kanban 的 task_events 为正式审计链 |
| 历史查询 | **Kanban + task_flow(只读)** | task_flow 作为补充查询通道，不写入 |
